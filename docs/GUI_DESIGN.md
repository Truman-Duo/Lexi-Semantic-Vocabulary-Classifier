# Lexi v3.0 GUI 设计报告

## 一、功能分析：用户要用 Lexi 做什么

Lexi 的核心价值链路有 3 个阶段：

```
阶段一：分类       阶段二：生成       阶段三：复习
输入文本    →    选词 → AI生成    →   间隔重复
→ 清洗/还原      → 风格约束          → 掌握追踪
→ 分类/排序      → 密度控制          → 练习巩固
→ 多格式输出     → 短文输出          → 统计反馈
```

当前已实现阶段一、二，阶段三为 P1 待开发。GUI 需要同时服务已实现功能和预留未来扩展。

### 阶段一的具体用户操作

| 步骤 | 用户要做什么 | 频率 | 重要性 |
|------|-------------|------|--------|
| 1. 选择输入文件 | 指定要分类的英文文本 | 每次必做 | 必须 |
| 2. 确认分类词库 | 使用内置词库或自定义 | 首次/偶尔 | 重要 |
| 3. 选择输出目录 | 指定结果保存位置 | 每次 | 中等 |
| 4. 勾选输出格式 | Markdown/JSON/CSV/HTML/Anki | 每次 | 重要 |
| 5. 执行分类 | 点击运行 | 每次必做 | 必须 |
| 6. 查看进度 | 了解处理到哪一步 | 运行时 | 重要 |
| 7. 查看结果 | 看日志确认成功 | 运行后 | 重要 |
| 8. 打开输出文件 | 直接访问结果文件 | 运行后 | 中等 |

### 阶段二的具体用户操作

| 步骤 | 用户要做什么 | 频率 | 重要性 |
|------|-------------|------|--------|
| 1. 配置 API | 填写 API Key / Base URL / Model | 首次 | 必须 |
| 2. 选择词汇数量 | 10/20/30/50 词 | 每次 | 重要 |
| 3. 选择策略 | balanced/top_frequency/random/stratified | 每次 | 中等 |
| 4. 选择密度 | short(3次/词)/medium(4次/词)/long(5次/词) | 每次 | 中等 |
| 5. 选择风格 | 无 / TOEFL / NewYorker 等模板 | 可选 | 低 |
| 6. 手动指定词汇 | 逗号分隔输入，留空自动选 | 可选 | 低 |
| 7. 执行生成 | 点击生成短文 | 每次必做 | 必须 |
| 8. 阅读短文 | 在日志区或外部文件查看结果 | 运行后 | 必须 |

### 阶段三的操作（P1 预留）

| 步骤 | 用户要做什么 |
|------|-------------|
| 查看词库状态 | 哪些词已掌握、哪些在学习 |
| 每日复习 | 系统自动筛选到期词汇 |
| 学习统计 | 各分类掌握率可视化 |

## 二、操作频率分析 → 视觉权重分配

按操作频率和重要性，将元素分为三个视觉层级：

| 层级 | 特征 | 包含元素 |
|------|------|---------|
| **一级（高频必做）** | 视觉突出、易于触达 | 输入文件选择、开始分类按钮、生成短文按钮 |
| **二级（每次/偶尔）** | 视觉清晰、不抢眼 | 输出格式、词汇数量、策略、密度、进度条 |
| **三级（首次/可选）** | 可折叠或收拢 | API配置、风格模板、指定词汇、分类词库路径 |

## 三、布局结构：从功能到空间

### 3.1 整体框架：两栏布局

```
┌──────────┬─────────────────────────────────────┐
│  侧栏     │  主内容区                              │
│  (200px) │                                      │
│          │  ┌─ 顶栏：当前页面标题 ──────────────┐ │
│  导航:    │  └────────────────────────────────┘ │
│  ·分类    │                                      │
│  ·生成    │  ┌─ 输入设置卡片 ──────────────────┐ │
│          │  └────────────────────────────────┘ │
│  ──────   │                                      │
│  ·词库    │  ┌─ 输出格式卡片 ──────────────────┐ │
│  ·复习    │  └────────────────────────────────┘ │
│  ·统计    │                                      │
│          │  ┌─ 开始分类 按钮 ──────────────────┐ │
│  ──────   │  └────────────────────────────────┘ │
│  ·API     │                                      │
│  ·风格    │  ┌─ 进度卡片 ──────────────────────┐ │
│          │  └────────────────────────────────┘ │
│          │                                      │
│          │  ┌─ AI短文生成卡片 ─────────────────┐ │
│          │  └────────────────────────────────┘ │
│          │                                      │
│          │  ┌─ 输出日志（弹性填充）────────────┐ │
│          │  └────────────────────────────────┘ │
│          │                                      │
│          │  ┌─ 底部栏：状态 + 快捷操作 ────────┐ │
│          │  └────────────────────────────────┘ │
└──────────┴─────────────────────────────────────┘
```

**设计理由：**

- **侧栏导航**：对应三个功能阶段（分类/生成/复习），P1 预留项标记"即将推出"，用户看到工具在成长
- **主内容区单页滚动**：分类和生成不是"先后两步"而是"同一流程的上下部分"——分类完成后立即进入生成，放在同一页减少切换
- **顶部标题**：当前页面名称，侧栏高亮对应项，双重确认用户位置
- **底部状态栏**：全局可见，不随内容滚动消失

### 3.2 卡片序列：从上到下的操作流

主内容区卡片从上到下排列，对应用户的自然操作顺序：

```
1. 输入设置     ← 用户先选择文件
2. 输出格式     ← 用户确认输出
3. [开始分类]   ← 执行
4. 进度         ← 反馈
5. AI短文生成   ← 下一步操作
6. 输出日志     ← 查看结果（弹性填充剩余空间）
```

**为什么输出日志在最后且弹性填充：**
- 日志是"查看"而非"操作"，放在最后不打断操作流
- 弹性填充 = 日志区域随窗口变大而变大，用户能看更多历史输出
- 按钮固定在日志上方，不随滚动消失

### 3.3 输入设置卡片：横向表单行

```
┌──────────────────────────────────────────────┐
│ 输入设置                                      │
│                                              │
│ 输入文件  [test.txt________________] [浏览]  │
│ 分类词库  [data/categories_full.json] [浏览]  │
│ 输出目录  [./_______________________] [浏览]  │
└──────────────────────────────────────────────┘
```

- 标签 70px 固定宽度，右对齐视觉整齐
- 输入框弹性填充，窗口缩放时自适应
- 浏览按钮统一右侧，符合"选择→确认"的认知顺序
- 分类词库和输出目录有默认值，减少用户操作

### 3.4 输出格式卡片：Chip 标签组

```
┌──────────────────────────────────────────────┐
│ 输出格式                                      │
│                                              │
│  ● Markdown  ● JSON  ● CSV  ● HTML浏览器  ○ Anki牌组 │
└──────────────────────────────────────────────┘
```

- Chip（圆角标签）替代复选框：视觉更轻、选中态更明显
- 默认全选（除 Anki），覆盖 90% 使用场景
- 横向排列无换行，减少垂直空间占用

### 3.5 运行按钮：全宽醒目

```
┌──────────────────────────────────────────────┐
│         ▶ 开始分类                             │
└──────────────────────────────────────────────┘
```

- 全宽 = 最容易点击，符合 Fitts 定律
- 蓝色强调 = 页面唯一的主操作色
- 卡片序列中的"分水岭"——按钮以上是配置，以下是反馈和下一步

### 3.6 进度卡片：精简行内

```
┌──────────────────────────────────────────────┐
│ 进度                                          │
│                                              │
│ ████████████████░░░░░░░░  67%                 │
│ 正在分类词汇 · 3,000 / 4,521                   │
└──────────────────────────────────────────────┘
```

- 4px 细进度条，不抢眼但清晰
- 百分比右侧对齐
- 状态文字一行，13px 小字，信息密度高

### 3.7 AI 短文生成卡片：网格 + 底行

```
┌──────────────────────────────────────────────┐
│ AI 短文生成                                    │
│                                              │
│ 词汇数量    选择策略     重复密度    风格模板   │
│ [20 ▾]     [各类别均衡▾] [中篇▾]   [无▾]      │
│                                              │
│ 指定词汇 [逗号分隔，留空则自动选择___________]  │
│                                     [生成短文]│
└──────────────────────────────────────────────┘
```

- 四个下拉框等宽网格排列，视觉对齐
- 标签在上、控件在下，符合阅读顺序
- "指定词汇"输入框弹性填充，生成按钮右对齐
- 绿色按钮 = 第二主操作色，与蓝色分类按钮区分

### 3.8 输出日志：等宽终端风格

```
┌──────────────────────────────────────────────┐
│ 输出日志                                      │
│                                              │
│ Lexi 词汇分类工具 v3.0                         │
│ ──────────────────────────────────            │
│ 清洗完成: 4,672 原始词 → 4,521 有效词           │
│ 词形还原完成: 4,521 唯一词形                    │
│ 分类完成: 覆盖率 100%                           │
│   词典精确匹配: 80.0%  派生模糊匹配: 1.1%        │
│   后缀规则: 4.0%      词性回退: 14.6%            │
│ ──────────────────────────────────            │
│ ✓ 分类完成。AI 短文生成已就绪。                  │
└──────────────────────────────────────────────┘
```

- 等宽字体 + 深色底 = "终端感"，技术用户熟悉
- 关键数字用蓝色高亮（`4,672`、`80.0%`）
- 成功信息用绿色
- 分隔线用灰色弱化，不争视觉

### 3.9 侧栏导航：功能分区

```
┌─────────────┐
│ L  Lexi     │  ← Logo + 名称
│             │
│ 核心功能     │  ← 分组标题（灰色小字）
```

- 上中下三段：核心功能 → 学习进度 → 设置

- 非活跃项：灰色、hover 变亮

### 3.10 底部状态栏：全局常驻

- 不随主内容区滚动，始终可见

## 四、配色体系：暗色主题的层次设计

### 4.1 设计原则

**不同功能区域用不同的"灰度"，而非单一黑色。**

| 原则 | 说明 |
|------|------|
| 底板最暗 | 不是纯黑，带微弱暖色调，模仿 IDE/编辑器的暗色环境 |
| 结构层略亮 | 侧栏、顶栏比底板亮一个台阶，形成功能分区 |
| 内容层再亮 | 卡片比结构层亮，承载实际内容，阅读舒适 |
| 输入框内凹 | 比卡片暗，产生"可填入"的内凹感 |
| hover 提亮 | 交互元素 hover 时提亮，即时反馈 |

### 4.2 色值阶梯

```
底板     #1a1b1e  ← 最深，页面背景
侧栏     #212327  ← 结构层，功能分区
卡片     #282a2f  ← 内容层，承载信息
输入框   #1e1f23  ← 内凹感，在卡片内
hover    #2f3137  ← 交互反馈
边框     #33353b  ← 分隔线
```

### 4.3 文字色阶梯

```
主文字   #e4e5e8  ← 标题、重要信息
正文     #c5c6ca  ← 一般内容文字
辅助     #8b8d94  ← 标签、说明文字
最弱     #5f6168  ← 占位符、禁用态
```

对比度检查：主文字/底板 = 12.4:1（远超 WCAG AAA 标准 7:1）

### 4.4 功能色

| 颜色 | 色值 | 用途 |
|------|------|------|
| 蓝色（主操作） | `#4a90d9` | 开始分类按钮、进度条、高亮数字、选中态 |
| 绿色（次操作） | `#5a9e6f` | 生成短文按钮、成功状态、状态指示点 |

只用两种功能色，避免"彩虹按钮"的混乱感。

### 4.5 字体

| 用途 | 字体栈 | 理由 |
|------|--------|------|
| 正文 | `PingFang SC, Microsoft YaHei, Segoe UI` | 中文优先，系统原生，无需加载 |
| 等宽 | `Cascadia Code, Consolas, SF Mono` | 日志/终端，对齐清晰 |

## 五、交互设计决策记录

| 决策 | 理由 |
|------|------|
| 单页布局（非多页/标签页） | 分类和生成是连续操作，单页减少切换 |
| 卡片序列（非网格/仪表盘） | 操作流是线性的，不是并行的 |
| 全宽主按钮 | 最常用操作，最大化点击区域 |
| Chip 替代 Checkbox | 视觉更轻，选中态对比度更高 |
| Progress Bar 细线（4px） | 进度不是主要信息，不抢注意力 |
| 日志区弹性填充 | 用户关心输出，越大越好 |
| 侧栏固定宽度 | 导航不需要弹性，固定更稳定 |
| 底部栏常驻 | 状态信息需要全局可见 |

## 六、P1 扩展预留

当前设计已为 P1（学习状态追踪 + 间隔复习）预留：

- 侧栏"学习进度"分组的 3 个导航项（我的词库/每日复习/学习统计）
- 导航项标记"即将推出"
- 主内容区通过侧栏切换即可切换页面，无需改布局

P1 实现时，只需：
1. 新增"我的词库"页面：词表 + 掌握状态标签
2. 新增"每日复习"页面：今日到期词汇列表
3. 新增"学习统计"页面：分类掌握率图表
